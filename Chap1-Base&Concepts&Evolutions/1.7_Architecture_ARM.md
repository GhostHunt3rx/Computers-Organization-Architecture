# 1.7 — Architecture ARM

> L'architecture **ARM** est la seconde famille de processeurs sur laquelle s'appuie le livre. Si l'x86 domine le marché des PC et serveurs, ARM domine massivement le marché des **systèmes embarqués et des appareils mobiles**. Comprendre ARM est aujourd'hui indispensable pour tout ingénieur travaillant dans l'embarqué.

---

## Évolution d'ARM

> **ARM** : Famille de microprocesseurs et microcontrôleurs basés sur les principes de conception **RISC**. ARM Holdings (Cambridge, Angleterre) conçoit les architectures mais **NE FABRIQUE PAS** de processeurs — elle licencie ses conceptions à des fabricants. ARM est probablement **l'architecture de processeur la plus utilisée dans le monde**, toutes catégories confondues.

### Les deux types de licences ARM

| Type de Licence | Ce que le client achète | Ce que le client fait | Exemple |
|---|---|---|---|
| **Licence de Processeur** | Droits d'utiliser la conception de processeur fournie par ARM dans leurs propres puces | Intègre le cœur ARM tel quel dans sa puce | Qualcomm Snapdragon, Apple A-series |
| **Licence d'Architecture** | Droits de concevoir leur propre processeur conforme à l'architecture ARM | Conçoit son propre cœur compatible ARM ISA | Apple M1/M2 (cœurs Firestorm/Icestorm) |

### Caractéristiques clés du succès d'ARM

- **Haute performance** tout en maintenant une **faible consommation d'énergie**
- **Petite taille de die** (*die size*) — moins de silicium nécessaire
- **Faible coût** grâce à la petite taille
- Architecture **idéale pour les applications embarquées**
- Présent dans tous les **smartphones Android et Apple** (iPod, iPhone, iPad)

## L'Architecture du Jeu d'Instructions ARM (ISA)

Le jeu d'instructions ARM est **hautement régulier**, conçu pour une implémentation efficace du processeur et une exécution efficace des programmes :

- Toutes les instructions ont une **longueur fixe de 32 bits**
- **Format régulier et prévisible** — facilite le décodage matériel
- Cette régularité rend l'ISA ARM adaptable sur une grande gamme de produits

### Thumb et Thumb-2

> **Thumb (et Thumb-2)** : Extension du jeu d'instructions ARM — un sous-ensemble réencodé en **16 bits**. Conçu pour améliorer les performances sur les bus de données 16-bit ou plus étroits, et pour permettre une **meilleure densité de code**. La version actuelle est Thumb-2, qui mélange instructions 16-bit et 32-bit.

| Critère | ARM (32-bit) | Thumb-2 (16/32-bit mixte) |
|---|---|---|
| **Largeur d'instruction** | Fixe : 32 bits | Variable : 16 ou 32 bits |
| **Densité de code** | Standard | Meilleure (~30% plus compact) |
| **Performances** | Maximum sur bus 32-bit | Meilleures sur bus 16-bit étroit |
| **Utilisation** | Cortex-A, Cortex-R | Cortex-M (uniquement Thumb-2) |
| **Registres** | 16 registres 32-bit | Accès limité aux registres dans les instructions 16-bit |

## La Famille Cortex — Les Trois Profils ARM

ARM Holdings licencie principalement la **famille Cortex** de microarchitecture de processeurs. Il existe trois profils Cortex, identifiés par les lettres **A, R et M** :

```
┌─────────────────────┬─────────────────────┬─────────────────────┐
│   CORTEX-A          │   CORTEX-R          │   CORTEX-M          │
│   (Application)     │   (Real-Time)       │   (Microcontroller) │
├─────────────────────┼─────────────────────┼─────────────────────┤
│ Applications        │ Applications temps  │ Microcontrôleurs    │
│ mobiles,            │ réel : ABS auto,    │ IoT, réseaux de     │
│ smartphones,        │ contrôleurs de      │ capteurs, élec-     │
│ tablettes, TV       │ stockage,           │ tronique auto,      │
│ intelligentes       │ impression          │ industrie           │
├─────────────────────┼─────────────────────┼─────────────────────┤
│ > 1 GHz             │ 2–4 GHz             │ MHz range           │
│ MMU requis          │ MPU (pas MMU)       │ MPU (pas MMU)       │
│ (Linux, Android)    │ Latence très faible │ Thumb-2 uniquement  │
│ ARM + Thumb-2       │ ARM + Thumb-2       │ Ultra basse consom. │
└─────────────────────┴─────────────────────┴─────────────────────┘
```

### Différence clé : MMU vs MPU

| Critère | MMU (Memory Management Unit) | MPU (Memory Protection Unit) |
|---|---|---|
| **Fonction principale** | Supporte la mémoire virtuelle et la pagination — traduit adresses virtuelles en physiques | Protège les régions mémoire critiques — empêche un programme d'accéder à la mémoire d'un autre |
| **Complexité** | Haute — nécessite OS complet | Simple — fonctionne sans OS ou avec OS minimal |
| **Usage** | Cortex-A (smartphones, PC embarqués) | Cortex-R et Cortex-M (microcontrôleurs) |
| **Mémoire virtuelle** | Oui — plusieurs programmes dans l'espace virtuel | Non — mémoire physique directe |
| **OS supportés** | Linux, Android, Windows, iOS | RTOS, FreeRTOS, bare metal |

## Les 7 versions du Cortex-M

Le Cortex-M est particulièrement important dans le livre car il sert d'exemple principal de processeur de système embarqué :

| Modèle | Architecture | Focus principal | Caractéristiques distinctives |
|---|---|---|---|
| **Cortex-M0** | ARMv6-M | Ultra bas coût, ultra basse puissance, simplicité | À partir de 12k portes logiques. Pour applications 8–16 bit. Le plus petit. |
| **Cortex-M0+** | ARMv6-M | Encore plus efficace en énergie que M0 | MPU optionnel, meilleure efficacité pipeline 2 étages. |
| **Cortex-M3** | ARMv7-M | Performance et efficacité énergétique, 16–32 bit | Debug et trace complets. Usage général. **Utilisé comme exemple principal dans le livre.** |
| **Cortex-M4** | ARMv7E-M | Traitement de signal numérique (DSP) | Toutes fonctionnalités M3 + instructions DSP + FPU optionnel. |
| **Cortex-M7** | ARMv7E-M | Hautes performances | Bus d'instructions et données 64-bit. Plus performant que M4. |
| **Cortex-M23** | ARMv8-M Baseline | M0+ avec sécurité | Similaire M0+, ajout instructions divide entier et fonctionnalités sécurité (TrustZone). |
| **Cortex-M33** | ARMv8-M Mainline | M4 avec sécurité | Similaire M4, ajout TrustZone pour sécurité renforcée. |

## Étude de cas : le Cortex-M3 et le microcontrôleur EFM32

Le livre utilise le **Cortex-M3** comme exemple principal d'un système embarqué. C'est le mieux adapté de tous les modèles ARM pour l'usage microcontrôleur à usage général.

### Structure du processeur Cortex-M3

```
┌───────────────────────────────────────────────────────────────────┐
│                    PROCESSEUR CORTEX-M3                           │
├──────────────────┬──────────────────────┬─────────────────────────┤
│  BUS MATRIX      │  LOGIQUE DEBUG       │   DAP                   │
│  (Connexion      │  (Halt, Step,        │   (Debug Access Port)   │
│   interfaces)    │   Breakpoints)       │                         │
├──────────────────┼──────────────────────┼─────────────────────────┤
│  NVIC            │  CŒUR ARM Cortex-M3  │   ETM                   │
│  (Interrupt      │  (ALU 32-bit,        │   (Embedded Trace)      │
│   Controller)    │   Décodeur Thumb,    │                         │
│                  │   Logique contrôle)  │                         │
├──────────────────┼──────────────────────┼─────────────────────────┤
│  INTERFACE ICODE │  INTERFACE SRAM/     │   MEMORY PROTECTION     │
│  (Fetch          │  Periph              │   UNIT                  │
│   instructions)  │  (Read/Write données)│   (Isolation processus) │
└──────────────────┴──────────────────────┴─────────────────────────┘
   ← Architecture Harvard : bus SÉPARÉS pour instructions (ICode)
                              et données (SRAM) →
```

### Architecture Harvard vs von Neumann dans le Cortex-M3

| Critère | Architecture von Neumann (IAS) | Architecture Harvard (Cortex-M3) |
|---|---|---|
| **Bus** | Un seul bus pour instructions ET données | Bus séparés pour instructions et pour données |
| **Accès simultané** | Impossible — un seul accès mémoire à la fois | Possible — lire une instruction ET des données simultanément |
| **Performance** | Goulot d'étranglement (*bottleneck*) mémoire | Meilleures performances grâce à la parallélisation |
| **Complexité** | Plus simple | Plus complexe (deux interfaces mémoire) |
| **Utilisé dans** | Ordinateurs classiques (conception originale) | Microcontrôleurs, processeurs DSP modernes |
| **Avantage principal** | Simplicité, flexibilité du code | Performances accrues par parallélisme instruction/données |

## Composantes du microcontrôleur EFM32 basé sur Cortex-M3

Le microcontrôleur **EFM32** de Silicon Labs illustre comment un Cortex-M3 est intégré dans un système complet sur puce. La puce comprend **10 domaines principaux** :

| Domaine | Contenu | Fonction |
|---|---|---|
| **Cœur et Mémoire** | Processeur Cortex-M3, SRAM 64 KB, Flash 64 KB, interface debug | Traitement, stockage programme (Flash non-volatile) et données temporaires (SRAM) |
| **Ports I/O Parallèles** | Broches GPIO, reset broches, interruptions externes | I/O numériques configurables pour tout type d'interface parallèle |
| **Interfaces Série** | USART, USB, UART, Low-Energy UART | Communication série standard pour tout type de périphérique |
| **Interfaces Analogiques** | Convertisseur A/N, Convertisseur N/A | Pont entre le monde analogique (capteurs, actionneurs) et numérique |
| **Timers et Déclencheurs** | Timer/compteur, timer temps réel, compteur pulse, watchdog | Mesure du temps, génération de signaux PWM, actions temporisées |
| **Gestion de l'Horloge** | Oscillateurs RC haute/basse fréq., oscillateurs cristal | Contrôle plusieurs horloges pour minimiser la consommation |
| **Gestion de l'Énergie** | Modes basse consommation, power-on reset, brownout detector | Gestion temps réel des besoins énergétiques, plusieurs modes sommeil |
| **Sécurité** | AES matériel (*Advanced Encryption Standard*) | Chiffrement et déchiffrement matériel pour communications sécurisées |
| **Bus 32-bit** | Interconnexion centrale | Relie tous les composants de la puce |
| **Bus Périphérique** | Réseau d'interconnexion entre périphériques | Permet aux modules périphériques de communiquer directement sans passer par le processeur |

### Applications du microcontrôleur EFM32

Ce microcontrôleur est commercialisé pour une grande variété de dispositifs, notamment :
- Compteurs d'énergie, gaz et eau intelligents
- Systèmes d'alarme et de sécurité
- Dispositifs d'automatisation industrielle
- Dispositifs d'automatisation domotique
- Accessoires intelligents (montres, trackers de fitness)
- Dispositifs de santé et fitness

## Comparaison : Microcontrôleur EFM32 vs Ordinateur multicœur

| Critère | Microcontrôleur EFM32 (Cortex-M3) | Ordinateur Multicœur (Core i7) |
|---|---|---|
| **Niveau du système** | Une seule puce = système complet | Une carte mère avec plusieurs puces distinctes |
| **Cache** | Aucun cache L1, L2 ou L3 | Caches L1, L2, L3 nombreux et hiérarchiques |
| **Raison du pas de cache** | Code en Flash locale : latence prévisible et constante | Mémoire externe : cache indispensable pour compenser la latence |
| **Mémoire** | Flash + SRAM intégrés (KB) | RAM externe (GB), SSD externe (TB) |
| **Consommation** | Microwatts en mode sommeil | Dizaines de watts en charge |
| **Coût** | Environ $1 pour le Cortex-M3 initial | Des centaines à milliers de dollars pour le processeur seul |
| **Performances** | Opérations simples déterministes | Milliards d'opérations par seconde, non déterministes |

---

> **Point clé :**
> - **ARM** est probablement l'architecture de processeur la plus utilisée au monde — présente dans presque tous les smartphones, tablettes et milliards d'appareils embarqués.
> - Le **modèle de licence ARM** (ne fabrique pas, licencie seulement) est un modèle économique unique qui a permis une adoption massive et diversifiée.
> - La **famille Cortex (A/R/M)** couvre tous les besoins : du smartphone à l'IoT minuscule.
> - Le **Cortex-M3** avec architecture Harvard illustre comment des décisions architecturales spécifiques (bus séparés) améliorent concrètement les performances embarquées.
