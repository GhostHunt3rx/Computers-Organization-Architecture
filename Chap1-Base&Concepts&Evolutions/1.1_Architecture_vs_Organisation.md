# 1.1 — Architecture et Organisation des Ordinateurs

> Cette section pose une distinction fondamentale qui traverse tout le domaine informatique. Comprendre la différence entre **architecture** et **organisation** est indispensable pour tout ingénieur ou développeur travaillant avec des systèmes informatiques. Ce sont deux concepts complémentaires mais distincts.

---

## Définitions fondamentales

### Architecture (ISA — Instruction Set Architecture)

Désigne **tous les attributs d'un système qui sont visibles par le programmeur**, c'est-à-dire ceux qui ont un impact direct sur l'exécution logique d'un programme.

L'ISA définit :
- Les **formats d'instructions**
- Les **codes opération** (opcodes)
- Les **registres**
- La **mémoire d'instructions et de données**
- L'**effet des instructions** exécutées sur les registres et la mémoire
- L'**algorithme de contrôle** d'exécution des instructions

### Organisation

Désigne **les unités opérationnelles et leurs interconnexions** qui réalisent concrètement les spécifications architecturales.

Elle inclut tous les détails matériels qui sont **transparents pour le programmeur** :
- Les signaux de contrôle internes
- Les interfaces entre l'ordinateur et ses périphériques
- La technologie mémoire utilisée

---

## Illustration pratique : l'instruction MULTIPLY

| DÉCISION ARCHITECTURALE | DÉCISION ORGANISATIONNELLE |
|---|---|
| Est-ce que l'ordinateur aura une instruction `MULTIPLY` ? | Comment implémenter `MULTIPLY` ? |
| → **Visible** par le programmeur | → **Invisible** pour le programmeur |
| Le programmeur écrit : `MULT R1, R2` | **Option A** : Unité matérielle dédiée (rapide mais cher) |
| Fait partie de l'ISA (interface visible) | **Option B** : Additions répétées via l'ALU (moins cher, plus lent) |

### Critères de la décision organisationnelle

La décision de choisir entre une unité dédiée ou des additions répétées dépend de plusieurs critères pratiques :
- La **fréquence d'utilisation** prévue de l'instruction de multiplication
- La **vitesse relative** des deux approches
- Le **coût et la taille physique** d'une unité de multiplication dédiée
- Les **contraintes de budget** et de consommation d'énergie

---

## Exemple historique : la Famille IBM System/370

Le livre cite l'**IBM System/370** comme exemple parfait illustrant ces concepts. Introduite en 1970, cette architecture a traversé des décennies tout en maintenant la compatibilité :

```
Évolution de l'IBM System/370 — Architecture stable, Organisation évolutive

  1970          1975          1985         1995        Aujourd'hui
Modèle 155   Mod. 115-168   303x, 308x   ES/9000    IBM z-Series (même ISA !)

←————— MÊME ARCHITECTURE (ISA) — les programmes fonctionnent sur toutes les générations —————→
                    Organisation (matériel) change à chaque génération
```

> **Point clé :**
> - L'investissement logiciel est **protégé** grâce à la stabilité de l'architecture.
> - Un client peut acheter un modèle moins cher au départ, puis monter en gamme **sans réécrire ses programmes**.
> - L'architecture peut être **améliorée** (ajout d'instructions) mais **jamais réduite** — c'est la compatibilité ascendante.

---

## Microprocesseurs : une relation plus étroite

Dans la classe des microordinateurs, la relation entre architecture et organisation est **beaucoup plus étroite** qu'avec les mainframes.

- Les changements technologiques influencent non seulement l'organisation, mais conduisent aussi à l'introduction d'architectures plus puissantes et plus complexes.
- Il y a **moins d'exigence de compatibilité** entre générations pour ces machines plus petites.

> **Exemple fascinant** : le RISC (*Reduced Instruction Set Computer*). La simplification de l'architecture (jeu d'instructions réduit) a permis d'optimiser l'organisation matérielle pour des performances maximales.

## Tableau récapitulatif

| Critère | Architecture (ISA) | Organisation |
|---|---|---|
| **Visibilité** | Visible par le programmeur | Transparente pour le programmeur |
| **Contenu** | Jeu d'instructions, registres, formats de données, I/O | Signaux de contrôle, technologies mémoire, interconnexions |
| **Stabilité** | Doit rester compatible sur de nombreuses années | Change avec les nouvelles technologies |
| **Exemple** | L'instruction `MULTIPLY` existe dans l'ISA | Implémentée par une ALU dédiée ou par additions répétées |
| **Impact** | Détermine **ce qu'on peut programmer** | Détermine les **performances et le coût** |
| **Niveau** | Abstrait / logique | Concret / matériel |
